‚Źļ This is an excellent design question. Let me break it down based on what
  exists and what would need to be built.

  Current State

  Gen mode entities are static ‚ÄĒ they have position, rotation, scale, material,
  and optional audio emitters, but no concept of "behavior over time." The only
  way to move something is gen_modify_entity (a discrete update, not continuous
  motion).

  How Behaviors Could Work

  The key insight is that behaviors need to be data, not code ‚ÄĒ so they're
  serializable, composable, and LLM-authorable.

  Behavior as a Component

  // A declarative behavior attached to an entity
  enum Behavior {
      // Motion primitives
      Orbit { center: String, radius: f32, speed: f32, axis: Vec3, phase: f32 },
      Spin { axis: Vec3, speed_deg_per_sec: f32 },
      Bob { axis: Vec3, amplitude: f32, frequency: f32 },
      Follow { target: String, speed: f32, offset: Vec3 },
      Oscillate { property: Property, min: f32, max: f32, freq: f32 },

      // Composition
      Parallel(Vec<Behavior>),    // All run simultaneously
      Sequence(Vec<TimedBehavior>), // Run in order with durations

      // Extensible
      Script { path: String, params: HashMap<String, Value> },
  }

  A Bevy system would tick every frame and apply these behaviors ‚ÄĒ Orbit
  computes position from time.elapsed(), Spin increments rotation, etc. The LLM
  would use tools like:

  gen_add_behavior(entity: "earth", behavior: "orbit", center: "sun", radius:
  5.0, speed: 1.0)
  gen_add_behavior(entity: "earth", behavior: "spin", axis: [0,1,0], speed:
  360.0)

  Tying to Data Structure

  Each GenEntity would gain a Vec<Behavior> component. Behaviors reference other
   entities by name (not Bevy Entity ID), matching the existing NameRegistry
  pattern. This means behaviors survive serialization:

  # behaviors.toml
  [[behaviors]]
  entity = "earth"
  type = "orbit"
  center = "sun"
  radius = 5.0
  speed = 1.0
  axis = [0, 1, 0]

  [[behaviors]]
  entity = "earth"
  type = "spin"
  axis = [0, 1, 0]
  speed = 360.0

  [[behaviors]]
  entity = "moon"
  type = "orbit"
  center = "earth"
  radius = 1.5
  speed = 3.0

  World-as-Skill Format

  This is where it gets really interesting. A "world skill" would be a
  self-contained directory:

  skills/solar-system/
  ‚Ēú‚ĒÄ‚ĒÄ SKILL.md              # Skill metadata + prompt instructions
  ‚Ēú‚ĒÄ‚ĒÄ scene.glb             # glTF geometry & materials (standard format)
  ‚Ēú‚ĒÄ‚ĒÄ world.toml            # World manifest (ties everything together)
  ‚Ēú‚ĒÄ‚ĒÄ behaviors.toml        # Declarative behavior definitions
  ‚Ēú‚ĒÄ‚ĒÄ audio.toml            # Ambience + spatial emitters
  ‚Ēú‚ĒÄ‚ĒÄ assets/
  ‚Ēā   ‚Ēú‚ĒÄ‚ĒÄ textures/         # Image textures referenced by glTF
  ‚Ēā   ‚Ēú‚ĒÄ‚ĒÄ sounds/           # Audio samples (.wav/.ogg)
  ‚Ēā   ‚ĒĒ‚ĒÄ‚ĒÄ scripts/          # Behavior scripts (Rhai or Lua)
  ‚ĒĒ‚ĒÄ‚ĒÄ README.md             # Human docs

  SKILL.md

  ---
  name: "Solar System"
  description: "Interactive solar system with orbiting planets"
  user-invocable: true
  metadata:
    emoji: "ūü™ź"
    type: "world"          # New: marks this as a gen world skill

    - contains: "solar system"
    - contains: "planets"

  volume = 0.3

  Extensibility Path

  The world.toml manifest is the key extensibility point. New asset types get
  new sections:

  # Future: interactive logic
  [[interactions]]
  entity = "info_panel"
  trigger = "click"           # or "proximity", "gaze"
  action = "show_text"
  text = "Earth: 3rd planet from the Sun"

  # Future: particle effects
  [[particles]]
  entity = "sun"
  type = "corona"
  color = [1.0, 0.9, 0.3]
  rate = 100

  # Future: custom scripts
  [[scripts]]
  entity = "comet"
  source = "scripts/comet_tail.rhai"
  params = { tail_length = 5.0 }

  For scripting, Rhai is the natural choice ‚ÄĒ it's pure Rust, sandboxable, and
  already used in the Bevy ecosystem. A script would look like:

  // scripts/comet_tail.rhai
  fn update(entity, dt) {
      let pos = entity.position;
      let vel = entity.get("velocity");
      entity.set_position(pos + vel * dt);
      // Adjust tail particles based on speed
      entity.set("tail_opacity", vel.length() / 10.0);
  }

  Implementation Strategy

  I'd suggest building this in layers:

  1. Behavior primitives ‚ÄĒ Add Orbit, Spin, Bob behaviors + gen_add_behavior
  tool + the Bevy tick system. This alone makes solar systems work.
  2. World serialization ‚ÄĒ gen_save_world / gen_load_world tools that write/read
   the skill directory format (scene.glb + behaviors.toml + audio.toml +
  SKILL.md).
  3. Script behaviors ‚ÄĒ Embed Rhai for custom per-entity logic beyond the
  built-in primitives.
  4. Interactions ‚ÄĒ Click/proximity triggers ‚Üí actions (show text, toggle
  visibility, play sound, run script).

  Want me to plan and implement any of these layers? Layer 1 (behavior
  primitives + tick system) would be the natural starting point ‚ÄĒ it's
  self-contained and immediately useful for things like the solar system
  example.
